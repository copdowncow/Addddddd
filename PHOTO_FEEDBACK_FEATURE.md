# Функционал оценки фото букета клиентом

## Описание
Добавлена система оценки фотоотчета магазина клиентом с возможностью запроса переделки и автоматическим подтверждением.

## Реализованные функции

### 1. Кнопки лайк/дизлайк при получении фото
- Когда магазин отправляет фото готового букета, клиент получает сообщение с двумя кнопками:
  - 👍 **Отлично** - клиент доволен фото
  - 👎 **Не устраивает** - клиент хочет переделку

### 2. Обработка лайка (одобрение)
- При нажатии "Отлично":
  - Кнопки исчезают
  - Клиент получает подтверждение: "Отлично! Магазин скоро доставит ваш заказ"
  - Магазин получает уведомление: "Клиент одобрил фото заказа #X! Можно доставлять"
  - В БД сохраняется `photo_approved = true`

### 3. Обработка дизлайка (недовольство)
- При нажатии "Не устраивает":
  - Кнопки исчезают
  - Клиенту предлагается написать причину: "Что не устраивает в фото?"
  - Клиент пишет текстовое сообщение с описанием проблемы
  - Отзыв отправляется магазину с текстом: "Клиенту не понравилось фото букета. Комментарий клиента: [текст]"
  - Магазину предлагается переделать букет и отправить новое фото
  - В БД сохраняется `photo_approved = false` и `photo_feedback = [текст]`

### 4. Повторная отправка фото магазином
- Магазин переделывает букет и отправляет новое фото через бот
- Система определяет, что это повторное фото (есть `photo_feedback`)
- Клиент снова получает фото с кнопками лайк/дизлайк
- Старый фидбек сбрасывается (`photo_approved = null`, `photo_feedback = null`)
- Процесс повторяется до одобрения клиента

### 5. Автоматическое одобрение (таймер)
- Если клиент не реагирует на фото в течение **30 минут**:
  - Фото автоматически одобряется (`photo_approved = true`)
  - Клиент получает уведомление: "Фото букета автоматически одобрено (прошло 30 минут)"
  - Магазин может продолжить доставку
- Проверка выполняется каждые **5 минут**

### 6. Отмена ввода причины
- Клиент может отменить ввод причины командой `/cancel`
- Состояние ожидания сбрасывается

## Изменения в базе данных

### Новые поля в таблице `orders`:
```sql
photo_approved BOOLEAN DEFAULT NULL
  -- true = клиент одобрил
  -- false = клиент отклонил
  -- null = клиент еще не ответил

photo_feedback TEXT DEFAULT NULL
  -- Текст причины при дизлайке

photo_approved_at TIMESTAMPTZ DEFAULT NULL
  -- Время одобрения фото

photo_feedback_at TIMESTAMPTZ DEFAULT NULL
  -- Время отправки фидбека
```

## Проверка работы

### Проверить запуск интервалов
В логах сервера должны быть строки:
```
⏱ Auto-confirm interval запущен (каждые 10 мин)
⏱ Shop order timer: reminder 5 min, auto-reject 10 min (poll 30s)
⏱ Auto-approve photo interval запущен (каждые 5 мин, таймаут 30 мин)
```

### Проверить БД
```sql
-- Проверить наличие новых полей
SELECT column_name, data_type, is_nullable 
FROM information_schema.columns 
WHERE table_name = 'orders' 
AND column_name IN ('photo_approved', 'photo_feedback', 'photo_approved_at', 'photo_feedback_at');
```

## Технические детали

### Таймауты
- **Автоодобрение фото**: 30 минут с момента `updated_at`
- **Проверка**: каждые 5 минут
- **Автоподтверждение заказа**: 5 часов (существующий функционал)

### Состояния заказа
- Фото отправляется когда `status = 'ready'`
- Фото-фидбек работает независимо от статуса доставки
- Клиент может дать фидбек даже если заказ уже в пути

### Безопасность
- Проверка владельца заказа через `ensureCustomerOwnsOrder`
- Ограничение длины фидбека: 1000 символов
- Дедупликация уведомлений

## Совместимость

### Обратная совместимость
- Старые заказы без новых полей работают как раньше
- Новые поля имеют `DEFAULT NULL`
- Если поля не заполнены - кнопки все равно показываются

### Миграция данных
Не требуется - все новые поля опциональные.

## Известные ограничения

1. Таймер работает только пока сервер запущен
2. Если сервер перезапустится - таймеры начнут отсчет заново
3. Фидбек хранится только последний (при повторном дизлайке старый перезаписывается)
